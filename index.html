내가 요청하는 것 — 필요한 필드만 (다이어트 JSON)

위 예시는 "전체 메뉴"고, 이건 내가 실제로 주문하는 것만 추린 최소 스펙이야. 우리 구조는 Pull 방식 — 각 팀이 DB에 저장해두면, 나는 SELECT로 아래 필드만 읽어옴. 각 팀은 이 표를 보고 "이 필드만 채워주면 된다"는 걸 알 수 있어. (= 인터페이스 계약)

💡 Pull 방식이란?

택배(Push)처럼 상대가 나한테 보내주는 게 아니라, 내가 우편함(DB)에서 필요한 것만 꺼내오는 방식. 그래서 각 팀은 "저장"만, 필요한 필드 선택은 "내가" 함. ❌ 표시 필드는 저장해도 내가 안 읽으니 안 만들어도 돼.

내 쪽 ERD — 읽는 테이블 & 쓰는 테이블

ERD = Entity Relationship Diagram(개체 관계도) — 어떤 테이블이 있고 각 테이블에 무슨 컬럼이 들어가는지, 서로 어떻게 연결되는지 보여주는 그림. 내 쪽은 입력 5개 테이블에서 읽어(SELECT) → 판단 → 내 결과 1개 테이블에 씀(INSERT).

Policy

판단 문턱값 — 공격형 (단일)

MVP는 공격형 하나로 간다 (안정형·균형형 제거). 공격형의 철학은 "확실해질 때까지 기다리지 말고, 방향이 잡히는 초기에 움직인다". 아래가 config.py의 POLICY 값이야.

🔥 공격형 — 기회면 움직임
  • 공시 sentiment / importance0.60 / 0.25
  • 뉴스 sentiment / grade / confirmed0.60 / 0.50 / 0.40
  • 뉴스 교차확인선택 (권장)
  • 기술 ml_prob_up0.50
  • 필요 합의 수 (4개 중)2개 이상
  • 매크로 (risk-off 시)conviction 감점
  • 최소 확신 (min_conviction)5.0

🧭 공격형이란 어떤 성향?

승률을 100%로 만들려고 완벽한 신호를 기다리면, 그땐 이미 주가가 다 오른 뒤야(늦은 진입). 공격형은 "방향이 잡히는 초기"에 자주 시도하고, 대신 틀린 건 손절(-5%)로 빠르게 끊어 손실을 제한해. 즉 승률보다 "기대값"을 노리는 전략.

📈 "기대값(Expected Value)"이란? 승률이 조금 낮아도 = (이길 확률 × 이길 때 수익)이 (질 확률 × 질 때 손실)보다 크면 장기적으로 이득이야. 공격형은 손실을 손절로 -5%에 묶어두고, 이기는 건 크게 먹어서(예: +15%) 기대값을 +로 만드는 걸 노려.

왜 이 숫자로 정했나 — 핵심 3개 (수치 이유 + 기대값)

① 필요 합의 수 = 2개 (4개 신호 중)

왜 2개? 4개 다 맞기를 기다리면 이미 뉴스 다 퍼지고 주가 오른 뒤야. 2개(예: 공시+기술)면 "방향이 잡히기 시작한 초기"라고 봐. 3개 이상은 너무 보수적(=안정형).
기대값: 진입 기회가 확 늘어남 → 시도 횟수 ↑. 대신 애매한 것도 섞이니까, 뒤에서 min_conviction과 손절이 걸러줌.

② 최소 확신 min_conviction = 5.0 (0~10 중)

왜 5.0? 딱 중간. "반반 이상 확신"이면 시도한다는 뜻. 8.0(안정형)으로 잡으면 웬만한 기회를 다 놓쳐서 공격형 취지가 죽어.
기대값: 중간 확신도 진입 → 기회 포착률 ↑. 틀린 것도 있지만 counter_scenario(손절 -5%)로 손실 제한 → 다수 시도 중 몇 개 크게 먹는 구조.

③ 뉴스 교차확인 = 선택 (필수 아님)

왜 필수 아님? Reuters+Bloomberg 교차확인까지 기다리면 이미 뉴스 확산 후 = 주가 반영됨. 공격형은 단독 출처라도 grade_score 0.50만 넘으면 시도해서 남보다 먼저 진입(선점).
기대값: 빠른 진입으로 초기 상승분 확보. 대신 루머 위험 ↑ → grade 0.50 최소선 + 매크로 감점으로 최소 방어.

📌 이 숫자들은 제안값(초기값)이야 — 테스트 데이터로 백테스트하면서 "기대값이 최대가 되는" 값으로 조정할 예정. 공격형이라도 무모하게 다 사는 게 아니라, "기회는 자주 잡되 손실은 짧게"가 핵심.

내가 처리하는 것 — 각 단계 기준과 이유

위 11단계를 하나씩, 어떤 기준으로 · 어떻게 처리하고 · 왜 그렇게 하는지(이유)까지 풀어놓은 거야. "얘가 무슨 근거로 매수/보류를 정하나"를 팀원 누구나 볼 수 있게.

1hard_block 체크 (공시 + 뉴스)
항목내용
기준공시 또는 뉴스의 hard_block == 1
처리GPT 호출 없이 즉시 "보류" (conviction = 0)
투자유형 차이없음 — 상폐 종목은 유형과 무관하게 무조건 차단
왜? 상폐·거래정지는 판단이 아니라 사실. LLM은 확률적이라 드물게 "그래도 분석해보자"고 할 수 있음 → 절대 규칙은 Python이 강제.
2매크로 위험 감점 (구 macro_veto)
⚠️ 설계 변경: 원래는 "risk-off면 무조건 보류(거부권)"였는데 → 폐지했음. 시장이 나빠도 개별 종목은 오를 수 있어서(시장 역행) 단정하면 좋은 기회를 놓침. 그래서 "거부권" → "conviction 감점"으로 바꿈. (팀원 제안 반영)
항목내용
기준regime == "risk-off"일 때, risk_score가 높을수록 conviction을 많이 깎음
처리보류 강제 ❌ → conviction에서 감점만 (진짜 강한 종목은 살아남음)
최종 필터감점된 conviction이 min_conviction(5.0) 미달이면 → 보류로 강등
매크로 감점표 (risk-off일 때, 제안값)
매크로_감점 = {   # risk-on이면 감점 0 (오히려 consensus +1)
     5: -0.5,
     6: -1.0,
     7: -1.5,
     8: -2.0,
     9: -3.0,
    10: -4.0,
}
conviction = conviction - 매크로_감점[risk_score]   # 시장 나쁠수록 많이 깎임
상황 (risk-off, 위험도 8 → 감점 -2.0)결과
강한 종목 conviction 9.59.5 − 2.0 = 7.5 → 문턱 넘으면 매수 가능 (기회 안 놓침)
애매한 종목 conviction 6.06.0 − 2.0 = 4.0 → 문턱 미달 → 자연히 보류
왜 이렇게? 시장이 나쁘면 좋은 종목도 같이 떨어지는 건 맞지만(태풍), "무조건 취소"는 시장 역행 기회를 놓침. 그래서 "강하게 눌러주기(감점)"로 바꿔서 — 정말 강한 종목만 눌려도 살아남고, 애매한 건 min_conviction 문턱에서 자연히 걸러지게 함. GPT에게도 regime·risk_score·이유를 넘겨서 "이 종목이 역행할 근거가 있나" 판단하게 함. 진짜 위험(상폐 등)은 여전히 hard_block이 지킴.
3신호별 매수 카운트 (+1 / 0)

각 신호를 아래 로직으로 +1(=1) / 0 판정. 합계 = signal_consensus (0~4).
★ 각 신호의 조건들은 전부 AND(그리고)로 묶여 있어 — 모든 조건을 만족해야 +1, 하나라도 못 넘으면 0.

3-1. 공시 판정
def 공시_카운트(disc):
    if disc.has_signal == 0:
        return 0                        # 오늘 공시 없음 → 카운트 없음
    # 공격형 문턱: sentiment 0.60, importance 0.25
    if disc.sentiment_score >= 0.60 and disc.importance >= 0.25:
        return 1                        # 호재이면서(AND) 충분히 큰 사건 → +1
    return 0

sentiment_score ≥0.600.9 ✓
importance ≥0.250.8 ✓
결과+1 (둘 다 충족)
3-2. 뉴스 판정 (조건 제일 많음 — 루머 방어)
def 뉴스_카운트(news):
    if news.has_signal == 0:
        return 0
    # 공격형: 3개 조건 (교차확인은 필수 아님 — 빠른 진입 위해)
    if (news.sentiment_score >= 0.60
        and news.grade_score >= 0.50          # 최소 신뢰 출처
        and news.confirmed_score >= 0.40):     # 확정 비율 40%+
        return 1
    return 0
조건 (전부 AND)공격형 문턱
sentiment_score ≥0.60
grade_score ≥0.50
confirmed_score ≥0.40
cross_source_confirmed필수 아님 (되면 확신 +0.15 보너스만)
AND의 의미: 위 3개 조건을 전부 넘어야 +1(=1). 하나라도 못 넘으면 0.
공격형이라 교차확인은 뺐어 — 교차확인까지 기다리면 이미 뉴스 퍼진 뒤(늦음)라, 단독 출처라도 grade 0.50 넘으면 시도. 대신 교차확인되면 확신에 +0.15 얹어줌.

def 기술_카운트(tech):

def 매크로_카운트(macro):

5GPT-4o 판단 — 뭘 보고 판단하나
★ 핵심 원칙: GPT는 숫자 계산을 다시 하지 않는다. 내가 계산 끝낸 요약본을 보고 "맥락 판단"만 한다.
GPT에게 넘어가는 것:
① 신호 4개 요약     ← ①에서 받은 값 (핵심 필드 + reason 텍스트)
② 신호별 판정 결과   ← 3번에서 계산 (공시+1, 뉴스0, 기술+1, 매크로+1 …)
③ consensus 합계    ← "4개 중 3개 매수 방향" + 매수 후보 여부
④ 과거 메모리       ← DB tb_memory_entries 테이블
⑤ 투자유형 정책      ← config.py POLICY

GPT가 내놓는 것:
side(매수/매도/보류) + conviction(0~10) + rationale + key_risk

④ 과거 메모리는 어디서?

Reviewer(파이프라인 ⑧)가 매 사이클 결과를 tb_memory_entries에 기록. 예: "2026-06-20 NVDA 매수 → -3% 손실. 원인: 실적 기대 과열에 늦게 진입". 나는 판단 전 그 ticker의 최근 5개를 꺼내 GPT 프롬프트에 넣음. → 같은 실수 반복 방지.

⑤ 정책(POLICY)은 어디서?

공격형 문턱 숫자(0.60, 5.0 등)는 config.py의 POLICY에 정의돼 있음. 실행 시 그 값을 꺼내 씀. (지금은 공격형 하나라 유형 분기 없음)

페르소나(투자 레전드 관점)는 MVP 2단계에서 추가 예정 — 지금은 안 넣음.

if signal.side == "매수" and signal.conviction < 5.0:
    signal.side = "보류"    # Python이 강제 (공격형 문턱 5.0)

Architecture · 코드 vs LLM

누가 뭘 결정하나 — 코드 게이트 + LLM 판단 (샌드위치 구조)

확실한 규칙은 코드가 강제하고, 맥락 판단은 LLM(GPT-4o)이 해. LLM은 앞문·뒷문(코드 게이트) 사이에 갇혀서 판단하는 구조야 — GPT가 아무리 "매수!"라 우겨도 게이트를 못 넘으면 소용없음.

① 코드 게이트 (앞문)

  • 🔒 hard_block → 즉시 보류
  • ✅ consensus 계산 (0~4)
  • 📊 매크로 위험 감점 계산
hard_block 걸리면 GPT까지 안 감
▶

② LLM 판단 (GPT-4o)

  • 신호 요약 + consensus + 매크로 맥락 + 메모리 + 정책을 종합
  • → conviction (0~10) + side(매수/보류)
매크로는 이제 여기서 "참고" (감점 + 맥락)
▶

③ 코드 게이트 (뒷문)

  • 🔒 min_conviction 검증
  • 매수인데 확신 < 문턱 → 보류 강등
GPT가 매수라 우겨도 여기서 최종 검문

확실하게 강제되는 규칙은 2개 (나머지는 감점·힌트·판단)

#규칙누가강제력하는 일
1hard_block코드🔒 강제상폐·거래정지 → 즉시 보류 (GPT 안 감)
2consensus 카운트코드✅ 계산매수 신호 몇 개인지 셈 (0~4)
3매크로 위험 감점코드📊 감점risk-off면 conviction 차감 (보류 강제 ❌)
—"보류 권장"코드→GPT🟡 권고consensus 미달이면 GPT에 힌트만 (강제 아님)
4conviction 판단GPT🔵 판단데이터 종합 → 0~10 점수 + 매수/보류
5min_conviction코드🔒 강제매수인데 확신 < 문턱 → 보류 강등
진짜 "강제"는 🔒 2개뿐 — hard_block(앞문) · min_conviction(뒷문). 매크로는 거부권을 뺐고(📊 감점으로), "보류 권장"은 GPT에 주는 참고 힌트일 뿐. 즉 확실히 막는 건 상폐 같은 객관 사실(hard_block)과 확신 미달(min_conviction)이고, 나머지는 GPT 판단 + 코드 감점으로 유연하게 처리.
◆conviction은 어떻게 나오나 — "공식 없음, GPT가 채점"

conviction = consensus×2 + 보너스 같은 사칙연산 공식이 없어. GPT-4o가 자료를 보고 채점하듯 0~10 점수를 매기는 거야. (면접관이 "음… 8점 정도?" 하는 것처럼)

GPT는 아무렇게나? → 아니, "채점 기준표"를 같이 줌
[너의 임무] 아래 자료를 보고 매수 확신도를 0~10으로 매겨라.

[채점 기준]
· consensus 4개 전원 + 교차확인 + 위험 낮음   → 8~10점
· consensus 3개 + 무난한 근거                → 6~8점
· consensus 2개 or 위험 요소 있음            → 4~6점
· consensus 낮거나 과거 실패 이력 있음        → 0~4점

[이번 종목 자료]
· NVDA · consensus 4 · 교차확인됨
· 위험: 인수 프리미엄 과다
· 과거 메모리: 지난달 NVDA 매수 -3% 물림

# → GPT: "consensus 4 + 교차확인이라 높은데, 과거에 물렸고
#         프리미엄 위험 있으니 만점은 아니고 → 8.5점"
conviction을 올리는 / 내리는 요인

⬆️ 올리는 요인

consensus 높음(4개 전원) · 신호 강함(sentiment 0.9) · 교차확인 됨 · 과거 이 패턴에서 수익 · 위험 요소 적음

⬇️ 내리는 요인

consensus 낮음(1~2개) · 신호 애매(0.65) · 교차확인 안 됨(루머 의심) · 과거 이 패턴에서 손실 · 위험 요소 많음

코드(나)가 conviction에 손대는 건 2가지뿐
conviction = GPT가_매긴_점수                # 예: 8.35
if cross_source_confirmed == 1:
    conviction += 0.15                      # ① 교차확인 보너스
if regime == "risk-off":
    conviction -= 매크로_감점[risk_score]      # ② 매크로 위험 감점
# → 최종 conviction
왜 공식 대신 GPT? 숫자 규칙만으론 "신호끼리 충돌"(공시는 호재인데 뉴스는 이미 반영됐다더라)이나 "맥락"을 못 잡음. 이건 사람 같은 판단이 필요해서 GPT에 맡김. 대신 명확한 규칙(🔒 게이트 2개: hard_block·min_conviction)은 코드가 강제해서 GPT가 폭주 못 하게 울타리를 침.
구분누가방식
딱딱한 규칙 (차단·문턱)코드(나)정해진 숫자 비교 (결정론적)
conviction 점수GPT-4o채점 기준 보고 종합 판단 (감각적)
+0.15 보너스 / 매크로 감점코드(나)교차확인 가산 · risk-off 차감

📌 스케일 주의 — 신호 점수(sentiment·grade 등)는 0~1(예: 0.8), conviction과 문턱은 0~10(예: 8.5 / 문턱 8.0·5.0). 서로 다른 스케일!

Output → Critic

Critic에게 보내는 것 — 설득 payload

내 판단 다음은 Risk Critic(반박 담당). 내 매수 결정에 일부러 반박해서 못 버티면 기각시킴. 그래서 "매수 8.5"만 던지면 안 되고 위험 인정 + 선제 반박 + 대응책 세트로 설득해야 통과됨.

{
  // ─── 기본 결정 (숫자) ───
  "ticker": "NVDA",
  "inv_type": "공격형",
  "side": "매수",
  "conviction": 8.5,
  "signal_consensus": 4,          // 4개 다 매수 → 강한 근거
  "cycle_id": 123,               // 원본 신호 찾는 열쇠

  // ─── 설득용 요약 (LLM 텍스트) ───
  "summary": "4개 신호 전원 매수 방향. 공시(인수)·뉴스(실적,교차확인)·기술(반등)·매크로(우호) 일치.",
  "bull_case": "① 대형 인수 공식 발표(프리미엄 30%) — 법적 확정 사실. ② 실적 서프라이즈를 Reuters·Bloomberg 교차확인. ③ RSI 34 과매도 반등권. ④ 시장 risk-on, VIX 16 안정.",

  // ─── 반박 대비 (Critic이 칠 만한 것 선제 방어) ───
  "key_risk": "인수 프리미엄 30% 과다 지불 → 단기 조정 가능",        // 위험 인정 (신뢰↑)
  "risk_rebuttal": "importance 0.8로 시장이 이미 긍정 반영 중, hard_block 없음, 공격형 기준(5.0) 충족(8.5)",
  "counter_scenarios": [                                          // 틀렸을 때 대응책
    "인수 무산 시 → 손절 라인 -5%",
    "시장 급변(risk-off 전환) 시 → 포지션 축소"

  "evidence": {

  "sizing_hint": {

    "conviction": 8.5,             // 이 확신에 근거
    "max_weight_cap": 0.05,        // 공격형 종목당 상한 5% 가이드
    "stop_loss_hint": -0.05,       // 손절 제안 -5%

항목역할왜 넣나
side·conviction·consensus결정 숫자기본 판단
summary한 줄 요약Critic이 빠르게 파악

원본 4개 JSON도 넘기나? — 아니, 요약만

✅ 넘기는 것

내 결정(side·conviction·consensus) · 요약·설득(summary·bull_case) · 핵심 근거 숫자(evidence) · 위험·반박·대응 · cycle_id + ticker (원본 찾는 열쇠)

❌ 안 넘기는 것

공시·뉴스·기술·매크로 원본 JSON 전체 — 이미 DB에 있으니 토큰 낭비. Critic이 의심되면 cycle_id + ticker로 직접 조회해서 검증.

핵심 설득 전략

❌ 나쁜 예 → 기각

"매수 8.5. 신호 좋음."
Critic: "근거가 뭐야? 위험은 생각했어?"

✅ 좋은 예 → 통과

"매수 8.5. 4개 신호 일치(bull_case), 위험은 프리미엄 과다지만 교차확인 + hard_block 없음으로 상쇄(rebuttal), 틀리면 -5% 손절(counter_scenario)."

역할 경계 — 어디까지가 내 몫?

항목내(Strategist) 몫Critic 몫
판단매수/보류 결정 + 근거(bull_case)그 판단에 반박·검증
위험내가 인지한 핵심 위험 1개 표시(key_risk)내가 놓친 위험 새로 발굴
원본 신호열쇠(cycle_id+ticker)만 넘김필요하면 직접 조회
반대 논리 총동원 (bear_case)❌ 안 함✅ Critic이 함
왜 bear_case(반대 근거)를 다 안 주나? 완전한 반대 논리를 짜는 건 Critic의 일이라 내가 대신 안 함. 나는 "내가 보기엔 이 위험이 있다"(key_risk)까지만 — 이건 "위험 알고도 판단했다"는 신뢰 표시지, Critic 일을 대신하는 게 아님. 그래서 지금 payload면 충분.
