{

  "as_of": "2026-07-02",                  // ✅ 분석 기준일 (FK) 🟡 → trade_date 이름 제안(미확정)
  "has_signal": 1,                       // ✅ 공시 있나 (0이면 뒤 컬럼 전부 null · 나는 =1인 행만 읽음)

  // ── B. 원문 정보 (감사·추적용) ──
  "filing_title": "NVIDIA to Acquire ...", // ❌ 원문 제목
  "filing_no": "0001-24-000123",          // ❌ SEC 접수번호(accession)
  "filed_at": "2026-07-02T08:55:00Z",     // ❌ 실제 제출 시각(look-ahead 방지)

  // ── C. 점수 (내 판단 핵심) ──
  "event_type": "ma",                     // 📄 공시 종류(11종) — ma = M&A(인수합병)
  "importance": 0.80,                    // ✅★ 사건 크기 (≥0.25이어야 신경씀)
  "risk_score": 0.30,                    // 📄 위험도(별개 축)
  "sentiment": "positive",                // ❌ 라벨 (아래 점수로 대체)
  "sentiment_score": 0.86,               // ✅★ 호재/악재 (≥0.70이면 호재)
  "reason": "35% 인수 프리미엄·전액 현금",   // 📄 점수 근거 (GPT 프롬프트)

  // ── D. 차단·부가 ──
  "hard_block": 0,                       // ✅★★ 1이면 즉시 보류 (제일 먼저 체크)
  "hard_block_reason": null,             // 📄 차단 이유
  "confidence": 0.9,                     // ❌ 분석 신뢰도 (거의 안 씀)
  "summary": "엔비디아, 대형 반도체사 인수 발표…", // 📄 3~4줄 요약 (GPT 참고)
  "keywords": ["인수", "반도체", "프리미엄"]  // ❌ 핵심 키워드 5개
}

② 뉴스 신호 (tb_news · 정본 25필드 전체)

{
  // ── A. 메타 (누가·언제) ──
  "ticker": "NVDA",                       // ✅ 어느 종목인지 (PK)
  "trade_date": "2026-07-02",             // ✅ 분석 기준일 (FK)
  "collected_at": "2026-07-02T09:30:00Z", // ✅ 스냅샷 생성 시각 (PK)
  "has_signal": 1,                       // ✅ 뉴스 있나 (0이면 뒤 컬럼 전부 null · 나는 =1인 행만 읽음)

  // ── B. 대표 기사 (신뢰가중 최상위 1건) ──
  "news_title": "Nvidia earnings beat Q2", // ❌ 대표 기사 제목 (표시용)
  "source": "reuters.com",                // ❌ 언론사 (grade_score로 대체)
  "published_at": "2026-07-02T07:10:00Z", // ❌ 발행 시각 (≠ 수집 collected_at)
  "ref": "https://reuters.com/...",       // ❌ 원문 링크 (감사용)
  "event_type": "earnings",               // 📄 뉴스 종류 (실적, 공시와 공유)

  "reason": "실적 서프라이즈",              // 📄 점수 근거 (GPT 프롬프트)
  "summary": "엔비디아 2분기 실적 예상 상회…", // 📄 3~4줄 요약 (GPT 참고)
  "keywords": ["실적", "데이터센터"],        // ❌ 핵심 키워드 5개

  // ── C. 전체 기사 신뢰가중 집계 ──
  "news_count": 3,                       // 📄 오늘 분석한 기사 총 건수
  "importance": 0.67,                    // ❌ 전체 뉴스 영향도(집계) → 무시
  "peak_importance": 0.8,                // ❌ 신뢰검증 강도 최댓값(협의중) → 무시
  "risk_score": 0.22,                    // 📄 하락 위험도(별개 축)
  "sentiment_score": 0.84,               // ✅★ 호재/악재 (≥0.70이면 호재)
  "sentiment": "positive",                // ❌ 라벨 (점수로 대체)

⚠️ 날짜 컬럼·사이클 연결 — 정본과 맞춘 주의사항 (팀 확인 필요)

  • 공시는 created_at, 뉴스는 collected_at — 이름이 다르다(정본 원문 그대로). 공시 쪽 "collected_at으로 통일"은 아직 제안일 뿐 미확정. 컬럼명이 달라 코드가 그대로 매칭 못 하니 팀 통일 필요.
  • 마찬가지로 공시 FK는 as_of, 뉴스 FK는 trade_date (공시 as_of→trade_date 이름 제안 중).
  • 업데이트 시간이 신호마다 다른데 어떻게 가져오나? → 정본 명시: "Strategist는 종목별 최신 행을 읽는다". 판단 시점에 종목마다 collected_at(공시는 created_at)이 가장 최근인 행 1개를 SELECT. (공시 1시간·뉴스 5분 주기로 append되니, 각자 최신 스냅샷을 집음)
  • 🔴 cycle_id는 입력 4개 테이블(공시·뉴스·기술·매크로)에 없다. PK는 전부 (ticker, collected_at/created_at). cycle_id는 내 결과표·Critic에만 있고, 생성 주체 미정(오케스트레이터 추정) — 정본도 "① cycle_id 생성 주체 미정"으로 🔴 미해결 표시. 그래서 입력 매칭은 (ticker + 최신 시각)으로 하고, cycle_id는 오케스트레이터가 사이클을 묶어줄 때 별도 매핑(팀 확인 중).

위 예시는 "전체 메뉴"고, 이건 내가 실제로 주문하는 것만 추린 최소 스펙이야. 우리 구조는 Pull 방식 — 각 팀이 DB에 저장해두면, 나는 SELECT로 아래 필드만 읽어옴. 각 팀은 이 표를 보고 "이 필드만 채워주면 된다"는 걸 알 수 있어. (= 인터페이스 계약)

💡 Pull 방식이란?

택배(Push)처럼 상대가 나한테 보내주는 게 아니라, 내가 우편함(DB)에서 필요한 것만 꺼내오는 방식. 그래서 각 팀은 "저장"만, 필요한 필드 선택은 "내가" 함. ❌ 표시 필드는 저장해도 내가 안 읽으니 안 만들어도 돼.

테이블 간 관계 (Relationships)

관계설명
입력 4개 → 나읽기(SELECT). 입력표엔 cycle_id가 없다 → ticker별 최신 행으로 매칭: 공시=(ticker, 최신 created_at) · 뉴스=(ticker, 최신 collected_at) · 기술=(ticker, trade_date) · 매크로=(as_of, 시장 1행). 판단 시점 기준 각자 가장 최근 스냅샷을 집음
나 → tb_strategist_signals쓰기(INSERT). 내 판단 결과 1행 저장 (원본 신호는 각 팀이 이미 저장했으니 나는 결과만)
tb_memory_entries → 나읽기. 같은 ticker의 최근 5개 과거 기록을 GPT 프롬프트에 넣음
공통 키 cycle_id오케스트레이터가 한 판단 사이클(08:30 실행)을 묶는 열쇠 — 내 결과(tb_strategist_signals)·Critic에만 존재. 🔴 입력 4개 테이블엔 cycle_id 컬럼이 없다 (공시=created_at·뉴스=collected_at·기술=trade_date·매크로=as_of가 PK). 사이클 연결은 오케스트레이터가 매핑 — 생성 주체 미정, 정본도 최우선 미해결로 표시. 내가 임의로 넣은 게 아니라, 아직 팀이 안 정한 것.
공통 키 ticker종목 식별 (tb_macro_signals만 없음 — 매크로는 시장 전체라서)
⚡ 실시간 스냅샷 (알파카)DB 테이블이 아니라 판단 순간 Alpaca API로 50개를 배치 조회 → 현재가·당일고저·거래량·52주고저를 내 결과 행에 저장. 시세는 금방 낡아서 시계열로 안 쌓음

📌 입력 테이블에는 다른 컬럼도 더 있지만, 여기엔 내가 실제로 SELECT하는 컬럼만 그렸어 (Pull 방식 = 필요한 것만 읽음). ❌ 필드는 위 "요청 스펙" 섹션 참고. 컬럼 타입은 PostgreSQL 기준 제안값이라 팀 합의로 조정 가능.

🔑 키(PK/FK) 한눈 정리 — "몇 개가 어디로 들어오나"

구분키설명
① 입력 읽을 때
(공시·뉴스·기술·매크로)		ticker + 최신 시각FK 4개를 따로 저장하는 게 아니라, 같은 ticker의 최신 행을 각 표에서 집어옴 (공시 created_at·뉴스 collected_at·기술 trade_date·매크로 as_of). 🔴 입력표엔 cycle_id가 없어서 이렇게 함.
② 내 결과표 PK
(tb_strategist_signals)		id BIGSERIAL (autoincrement)맞아 — 그냥 autoincrement로 만들면 돼. 행마다 DB가 자동으로 1씩 증가시키는 고유번호. 이게 곧 Critic이 참조하는 signal_id.
③ 내 결과표 FKticker + cycle_id내 행이 어느 종목·어느 사이클 판단인지. ticker 외엔 cycle_id 하나만 — 4개가 아니라 2개. (cycle_id 생성 주체는 아직 미정 🔴)
④ Critic으로 넘길 때signal_id(=내 id) + cycle_id정본 릴레이 키. Critic(tb_critic_verdict)은 signal_id FK로 내 행을 가리키고, 같은 cycle_id로 사이클을 확인.
정리: 질문하신 "4개 FK"는 아니에요 — 입력 4개 표는 ticker + 최신시각으로 그때그때 읽어오는 거고, 내 테이블에 저장하는 키는 id(PK·autoincrement) + ticker + cycle_id 뿐. Critic과는 signal_id(=id) + cycle_id로 이어져요. 그래서 PK는 그냥 autoincrement(BIGSERIAL id)로 만들면 정본과 일치합니다.

🧭 공격형이란 어떤 성향?

승률을 100%로 만들려고 완벽한 신호를 기다리면, 그땐 이미 주가가 다 오른 뒤야(늦은 진입). 공격형은 "방향이 잡히는 초기"에 자주 시도하고, 대신 틀린 건 손절(-5%)로 빠르게 끊어 손실을 제한해. 즉 승률보다 "기대값"을 노리는 전략.

📈 "기대값(Expected Value)"이란? 승률이 조금 낮아도 = (이길 확률 × 이길 때 수익)이 (질 확률 × 질 때 손실)보다 크면 장기적으로 이득이야. 공격형은 손실을 손절로 -5%에 묶어두고, 이기는 건 크게 먹어서(예: +15%) 기대값을 +로 만드는 걸 노려.

기대값: 다수 시도 중 몇 개 크게 → 손절 규율이 생명

어떻게 고르나? 이건 "정답"이 아니라 실험 대상이야. 과거 데이터로 A·B·C를 각각 돌려서 누적 수익(기대값)이 최대인 프리셋을 채택하면 돼. config.py에 PRESET = "B"처럼 두고 바꿔가며 테스트하는 걸 추천. 시작은 B(표준)부터.
왜 이 숫자로 정했나 — 표준(B) 핵심 3개 (수치 이유 + 기대값)

① 필요 합의 수 = 2개 (3개 신호 중 · 매크로 제외)

왜 2개? 3개(공시·뉴스·기술) 다 맞기를 기다리면 이미 뉴스 다 퍼지고 주가 오른 뒤야. 2개(예: 공시+기술)면 "방향이 잡히기 시작한 초기"라고 봐. (매크로는 표에서 빠지고 감점으로만 반영)
기대값: 진입 기회가 확 늘어남 → 시도 횟수 ↑. 대신 애매한 것도 섞이니까, 뒤에서 min_conviction과 손절이 걸러줌.

② 최소 확신 min_conviction = 0.6 (0~1 중)

왜 0.6? 중간(0.5)보다 살짝 위 — "반반보다 확실할 때만" 시도한다는 뜻. 0.8(안정형)으로 잡으면 웬만한 기회를 다 놓쳐서 공격형 취지가 죽어.
기대값: 중간 확신도 진입 → 기회 포착률 ↑. 틀린 것도 있지만 counter_scenario(손절 -5%)로 손실 제한 → 다수 시도 중 몇 개 크게 먹는 구조.

③ 뉴스 확인 보너스 = 선택 (필수 아님)

왜 필수 아님? 공식 공시(disclosure_ref)로 확인될 때까지 기다리면 이미 뉴스 확산 후 = 주가 반영됨. 게다가 disclosure_ref는 거의 안 붙어서 관문으로 쓰면 대부분 탈락해버려. 그래서 공격형은 source_trust 0.55 + grade_score 0.55만 넘으면 시도하고, disclosure_ref가 있으면 보너스만 얹어 준다.
기대값: 빠른 진입으로 초기 상승분 확보. 대신 루머 위험 ↑ → source_trust·grade 최소선 + 매크로 감점으로 최소 방어.

각 문턱값을 왜 그 숫자로? — 개별 근거

위 3개(합의·확신·확인)는 전략 축이고, 아래는 신호별 점수 문턱을 왜 그 수치로 잡았는지야. 전부 제안값 — 백테스트로 조정.

문턱값(표준 B)왜 이 값인가 (근거)
sentiment_score ≥0.700~1에서 0.5=중립. 0.70은 "확실히 호재 쪽"으로 기운 지점 — 0.5~0.65의 애매한 호재는 노이즈가 많아 제외. 너무 높이면(0.85+) 대형 호재만 남아 기회를 놓침 → 공격형은 0.70.
importance ≥ (공시)0.25사건 크기 하한. 0.25 미만은 "경미한 정정·소액 계약"급이라 주가를 못 흔듦 → 무시. 낮게(0.25) 잡아 어지간한 사건은 다 후보로 올리고, 방향(sentiment)으로 거른다.
grade_score ≥ (뉴스)0.55출처 도메인 등급(로이터 1.0·미등록 0.6·차단 0). 0.55는 미등록(0.6) 언저리까지는 허용하되 차단(0)·저급은 배제 — 공격형이라 로이터급만 기다리지 않고 선점.
source_trust ≥ (뉴스)0.55LLM이 내용까지 보고 매긴 신뢰도. 삭제된 confirmed_score 자리를 대체하는 관문. 0.55는 "반반보다 확실히 믿을 만" 지점 — grade(코드)와 trust(LLM) 두 축을 함께 넘어야 루머·펌프를 걸러냄.
ml_prob_up > (기술)0.60ML 상승확률. 0.5=동전던지기라 그보다 확실히 위(0.60)일 때만 기술 +1. 0.60은 "10번 중 6번 이상 오름" — 승률 우위가 생기는 최소선.
공통 논리: 공격형은 단일 문턱을 무리하게 높이지 않는 대신, 여러 축(방향·크기·출처·기술)을 각각 최소선만 넘게 하고 2개 이상 합의(consensus≥2)로 교차검증한다. 즉 "하나가 아주 세다"보다 "여럿이 방향이 맞다"를 본다. 그래서 개별 문턱은 중간값보다 살짝 위(0.55~0.70)에 모여 있다.

📌 이 숫자들은 제안값(초기값)이야 — 테스트 데이터로 백테스트하면서 "기대값이 최대가 되는" 값으로 조정할 예정. 공격형이라도 무모하게 다 사는 게 아니라, "기회는 자주 잡되 손실은 짧게"가 핵심.

강한 종목 conviction 0.950.95 − 0.20 = 0.75 → 문턱 넘으면 매수 가능 (기회 안 놓침)
애매한 종목 conviction 0.60.6 − 0.20 = 0.40 → 문턱 미달 → 자연히 보류
왜 이렇게? 시장이 나쁘면 좋은 종목도 같이 떨어지는 건 맞지만(태풍), "무조건 취소"는 시장 역행 기회를 놓침. 그래서 "강하게 눌러주기(감점)"로 바꿔서 — 정말 강한 종목만 눌려도 살아남고, 애매한 건 min_conviction 문턱에서 자연히 걸러지게 함. GPT에게도 regime·risk_score·이유를 넘겨서 "이 종목이 역행할 근거가 있나" 판단하게 함. 진짜 위험(상폐 등)은 여전히 hard_block이 지킴.
3신호별 매수 카운트 (+1 / 0)

각 신호를 아래 로직으로 +1(=1) / 0 판정. 합계 = signal_consensus (0~3 · 매크로 제외).
★ 각 신호의 조건들은 전부 AND(그리고)로 묶여 있어 — 모든 조건을 만족해야 +1, 하나라도 못 넘으면 0.

🔎 if disc.has_signal은 어디서·왜 검증하나 — 검증 순서

어디서: 내 preprocess.py의 각 카운트 함수(공시_카운트·뉴스_카운트) 맨 첫 줄에서 검증해. 아래 코드의 if …has_signal == 0: return 0이 바로 그 지점.

왜 필요한가: 정보분석팀은 공시·뉴스가 없는 종목도 행을 만들어(항상 50행/사이클로 균일) — 대신 그 행은 has_signal=0이고 뒤 컬럼이 전부 null이야. null인 채로 sentiment_score ≥ 0.70 같은 비교를 하면 에러나 오판이 나지. 그래서 "빈 행이냐"부터 걸러 0(기권)으로 즉시 반환해.

전체 검증 순서(한 종목): ① has_signal==0 → 기권(스킵)  →  ② hard_block==1 → 즉시 보류(STEP 1)  →  ③ 점수 문턱(sentiment·importance·grade·source_trust) 통과 시 +1. ①②는 안전장치(null·차단 방어), ③이 실제 표 세기.

Architecture · 코드 vs LLM

누가 뭘 결정하나 — 코드 게이트 + LLM 판단 (샌드위치 구조)

확실한 규칙은 코드가 강제하고, 맥락 판단은 LLM(GPT-4o)이 해. LLM은 앞문·뒷문(코드 게이트) 사이에 갇혀서 판단하는 구조야 — GPT가 아무리 "매수!"라 우겨도 게이트를 못 넘으면 소용없음.

① 코드 게이트 (앞문)

  • 🔒 hard_block → 즉시 보류
  • ✅ consensus 계산 (0~3 · 매크로 제외)
  • 📊 매크로 위험 감점 계산
hard_block 걸리면 GPT까지 안 감
▶

② LLM 판단 (GPT-4o)

  • 신호 요약 + consensus + 매크로 맥락 + 메모리 + 정책을 종합
  • → conviction (0~1) + side(매수/보류)
매크로는 이제 여기서 "참고" (감점 + 맥락)
▶

③ 코드 게이트 (뒷문)

  • 🔒 min_conviction 검증
  • 매수인데 확신 < 문턱 → 보류 강등
GPT가 매수라 우겨도 여기서 최종 검문

확실하게 강제되는 규칙은 2개 (나머지는 감점·힌트·판단)

#규칙누가강제력하는 일
1hard_block코드🔒 강제상폐·거래정지 → 즉시 보류 (GPT 안 감)
2consensus 카운트코드✅ 계산매수 신호 몇 개인지 셈 (0~3 · 매크로 제외)
3매크로 위험 감점코드📊 감점risk-off면 conviction 차감 (보류 강제 ❌)
—"보류 권장"코드→GPT🟡 권고consensus 미달이면 GPT에 힌트만 (강제 아님)
4conviction 판단GPT🔵 판단데이터 종합 → 0~1 점수 + 매수/보류
5min_conviction코드🔒 강제매수인데 확신 < 문턱 → 보류 강등
매수를 "무조건" 막는 건 hard_block 하나뿐. hard_block(상폐·거래정지)은 어떤 상황이든 즉시 보류. 반면 min_conviction은 "조건부" — 확신 점수가 문턱(0.6) 미달일 때만 강등하고, 매크로 감점은 아예 못 막고 점수만 깎아. 즉 코드가 강제로 개입하는 지점은 hard_block(무조건) + min_conviction(조건부) 2개고, 매크로·"보류 권장"은 막는 게 아니라 감점·힌트로 유연하게 반영.
◆conviction은 어떻게 나오나 — "공식 없음, GPT가 채점"

conviction = consensus×2 + 보너스 같은 사칙연산 공식이 없어. GPT-4o가 자료를 보고 채점하듯 0~1 점수를 매기는 거야. (면접관이 "음… 8점 정도?" 하는 것처럼)

GPT는 아무렇게나? → 아니, "채점 기준표"를 같이 줌

Trigger · Critic 연결

Critic 연결 방식 — 폴링 대신 트리거

Strategist 판단이 끝났을 때 Critic이 그 결과를 어떻게 넘겨받는가. 핵심은 Critic이 계속 확인(폴링)하지 않고, Strategist가 끝나면 넘겨주는(트리거) 방식.

❌ 폴링 (비효율)

Critic이 "Strategist 끝났나?"를 계속 물어봄 → 자원 낭비 + 반응 지연.
비유: 택배 올 때까지 문 앞에서 계속 확인 🚪

✅ 트리거 (권장)

Strategist가 완료하면 Critic을 부름(호출)/알림 → 즉시 반응, 낭비 없음.
비유: 초인종 — 도착하면 알림 🔔

현재 설계 (1차 MVP) — 직접 호출 트리거

오케스트레이터가 Strategist 완료 직후 매수 시그널만 즉시 Critic에 넘긴다. Critic은 폴링하지 않는다.

# pipeline/orchestrator.py
signal  = fundmanager.propose_signal(bundle)   # ⑤ 끝나면
verdict = critic.critique(signal)              # ⑥ 즉시 호출 (트리거!)

📌 매수만 Critic으로(split_for_routing) · 보류는 회고용 저장만 → Critic 부하도 줄인다.

2차 확장 — 이벤트 기반

종목이 많거나 비동기 처리가 필요하면: 메시지 큐 / DB의 LISTEN·NOTIFY 등으로 Strategist 저장 시 이벤트 발행 → Critic 구독. 지금은 단방향 1패스라 직접 호출로 충분.
