내가 요청하는 것 — 필요한 필드만 (다이어트 JSON)

위 예시는 "전체 메뉴"고, 이건 내가 실제로 주문하는 것만 추린 최소 스펙이야. 우리 구조는 Pull 방식 — 각 팀이 DB에 저장해두면, 나는 SELECT로 아래 필드만 읽어옴. 각 팀은 이 표를 보고 "이 필드만 채워주면 된다"는 걸 알 수 있어. (= 인터페이스 계약)

💡 Pull 방식이란?

택배(Push)처럼 상대가 나한테 보내주는 게 아니라, 내가 우편함(DB)에서 필요한 것만 꺼내오는 방식. 그래서 각 팀은 "저장"만, 필요한 필드 선택은 "내가" 함. ❌ 표시 필드는 저장해도 내가 안 읽으니 안 만들어도 돼.

내 쪽 ERD — 읽는 테이블 & 쓰는 테이블

ERD = Entity Relationship Diagram(개체 관계도) — 어떤 테이블이 있고 각 테이블에 무슨 컬럼이 들어가는지, 서로 어떻게 연결되는지 보여주는 그림. 내 쪽은 입력 5개 테이블에서 읽어(SELECT) → 판단 → 내 결과 1개 테이블에 씀(INSERT).

tb_memory_entries → 나읽기. 같은 ticker의 최근 5개 과거 기록을 GPT 프롬프트에 넣음
공통 키 cycle_id한 번의 판단 사이클(08:30 실행)을 묶는 열쇠 — 생성 주체는 오케스트레이터, 팀 확인 중
공통 키 ticker종목 식별 (tb_macro_signals만 없음 — 매크로는 시장 전체라서)

📌 입력 테이블에는 다른 컬럼도 더 있지만, 여기엔 내가 실제로 SELECT하는 컬럼만 그렸어 (Pull 방식 = 필요한 것만 읽음). ❌ 필드는 위 "요청 스펙" 섹션 참고. 컬럼 타입은 PostgreSQL 기준 제안값이라 팀 합의로 조정 가능.

3신호별 매수 카운트 (+1 / 0)

각 신호를 아래 로직으로 +1(=1) / 0 판정. 합계 = signal_consensus (0~4).
★ 각 신호의 조건들은 전부 AND(그리고)로 묶여 있어 — 모든 조건을 만족해야 +1, 하나라도 못 넘으면 0.

3-1. 공시 판정
def 공시_카운트(disc, inv_type):
    if disc.has_signal == 0:
        return 0                        # 오늘 공시 없음 → 카운트 없음
    문턱 = {"안정형": {"sent": 0.70, "imp": 0.40},
           "공격형": {"sent": 0.60, "imp": 0.25}}[inv_type]
    if disc.sentiment_score >= 문턱["sent"] and disc.importance >= 문턱["imp"]:
        return 1                        # 호재이면서(AND) 충분히 큰 사건 → +1
    return 0

importance 0.350.40 미달 ✗0.25 충족 ✓

def 뉴스_카운트(news, inv_type):
    if news.has_signal == 0:
        return 0
    if inv_type == "안정형":
        if (news.sentiment_score >= 0.70
            and news.grade_score >= 0.70          # 최상급 출처만
            and news.confirmed_score >= 0.60      # 확정 비율 60%+
            and news.cross_source_confirmed == 1): # 교차확인 필수!
            return 1
    else:  # 공격형
        if (news.sentiment_score >= 0.60
            and news.grade_score >= 0.50
            and news.confirmed_score >= 0.40):    # 교차확인은 필수 아님
            return 1
    return 0
조건 (전부 AND)안정형공격형
sentiment_score ≥0.700.60
grade_score ≥0.700.50
confirmed_score ≥0.600.40
cross_source_confirmed= 1 필수없어도 됨
AND의 의미: 안정형 뉴스는 위 4개 조건을 전부 넘어야 +1(=1). 예를 들어 sentiment·grade·confirmed 다 좋아도 cross_source_confirmed가 0이면 → 0(카운트 안 됨). "하나라도 빠지면 인정 안 함"이 루머 방어의 핵심.

5GPT-4o 판단 — 뭘 보고 판단하나
★ 핵심 원칙: GPT는 숫자 계산을 다시 하지 않는다. 내가 계산 끝낸 요약본을 보고 "맥락 판단"만 한다.
GPT에게 넘어가는 것:
① 신호 4개 요약     ← ①에서 받은 값 (핵심 필드 + reason 텍스트)
② 신호별 판정 결과   ← 3번에서 계산 (공시+1, 뉴스0, 기술+1, 매크로+1 …)
③ consensus 합계    ← "4개 중 3개 매수 방향" + 매수 후보 여부
④ 과거 메모리       ← DB tb_memory_entries 테이블
⑤ 투자유형 정책      ← config.py POLICY

GPT가 내놓는 것:
side(매수/매도/보류) + conviction(0~10) + rationale + key_risk

④ 과거 메모리는 어디서?

Reviewer(파이프라인 ⑧)가 매 사이클 결과를 tb_memory_entries에 기록. 예: "2026-06-20 NVDA 매수 → -3% 손실. 원인: 실적 기대 과열에 늦게 진입". 나는 판단 전 그 ticker의 최근 5개를 꺼내 GPT 프롬프트에 넣음. → 같은 실수 반복 방지.

⑤ 투자유형 정책은 어디서?

사용자가 처음 "안정형/공격형" 설정 → DB에 저장. 문턱 숫자(0.70, 8.0 등)는 config.py의 POLICY에 정의. 실행 시 DB에서 유형을 읽고 → config.py에서 그 유형 문턱 세트를 꺼내 씀.

페르소나(투자 레전드 관점)는 MVP 2단계에서 추가 예정 — 지금은 안 넣음.

Critic에게 보내는 것 — 설득 payload

내 판단 다음은 Risk Critic(반박 담당). 내 매수 결정에 일부러 반박해서 못 버티면 기각시킴. 그래서 "매수 8.5"만 던지면 안 되고 위험 인정 + 선제 반박 + 대응책 세트로 설득해야 통과됨.

원본 4개 JSON도 넘기나? — 아니, 요약만

✅ 넘기는 것

내 결정(side·conviction·consensus) · 요약·설득(summary·bull_case) · 핵심 근거 숫자(evidence) · 위험·반박·대응 · cycle_id + ticker (원본 찾는 열쇠)

❌ 안 넘기는 것

공시·뉴스·기술·매크로 원본 JSON 전체 — 이미 DB에 있으니 토큰 낭비. Critic이 의심되면 cycle_id + ticker로 직접 조회해서 검증.
