내 위치의 의미

4개 분석팀의 신호가 처음으로 "결정"으로 바뀌는 지점. 내 출력이 부실하면 뒤의 모든 단계가 흔들린다.

왜 멀티 에이전트?

한 AI가 다 하면 검증이 없다. 판단(나) → 반박(Critic) → 크기 결정(PM)으로 역할을 쪼개 서로 견제하는 구조.

내 담당 범위

STEP 2~7 — 신호 읽기 → 전처리 → 정책 적용 → 메모리 로딩 → GPT-4o 판단 → 검증·저장·전달.

Design Decisions

설계하면서 했던 고민들 — 문제 · 고민 · 결정

단순히 "만든 것"보다, 왜 그렇게 만들었는지가 이 프로젝트에서 배운 전부라고 생각한다. 실제로 부딪히고 뒤집었던 결정 6가지.

결정 01 · 데이터 흐름

Push가 아니라 Pull — "택배 말고 우편함"

문제
4개 분석팀의 신호를 어떻게 받을까? 각 팀이 나한테 보내주나(Push), 내가 가져오나(Pull)?
고민
Push는 팀마다 전송 코드가 필요하고, 나 하나 바뀌면 4팀이 다 고쳐야 한다. 결합이 너무 세다.
결정
각 팀은 DB에 저장만, 나는 SELECT로 필요한 필드만 읽는 Pull 방식. 인터페이스 계약(어떤 필드가 필요한지)을 문서로 명시해서 "저장 형식"만 합의하면 서로 독립적으로 개발 가능.
결정 02 · 코드 vs LLM

규칙은 코드가, 맥락은 LLM이 — 샌드위치 구조

문제
LLM은 확률적이라 상폐 종목도 "그래도 분석해보자"고 할 수 있다. 반대로 숫자 규칙만으론 "공시는 호재인데 뉴스는 이미 반영됐다더라" 같은 맥락을 못 잡는다.
고민
전부 코드로 하면 유연성이 없고, 전부 LLM에 맡기면 폭주를 못 막는다. 경계를 어디에 긋나?
결정
앞문(hard_block) → LLM 판단 → 뒷문(min_conviction)의 샌드위치 구조. 절대 규칙 2개는 Python이 강제하고, 그 사이에서 GPT-4o가 채점 기준표(루브릭)를 보고 conviction(0~10)을 매긴다. LLM은 울타리 안에서만 판단한다.
결정 03 · 피드백으로 뒤집은 설계

매크로 "거부권" 폐지 → 수치화된 "감점"으로

문제
처음 설계는 "시장이 risk-off + 위험도 높으면 무조건 보류(거부권)". 그런데 팀원이 지적했다 — "경기가 나빠도 오르는 종목이 있는데, 매크로 하나가 거부권을 가질 정도인가?"
고민
맞는 말이었다. 시장 역행 종목(방어주, 개별 호재)은 실제로 존재한다. 거부권은 너무 뭉툭한 칼이라 좋은 기회까지 잘라버린다.
결정
거부권을 없애고, 위험도만큼 conviction을 깎는 감점제로 전환. 진짜 강한 종목(9.5점)은 감점(-2.0)돼도 살아남고, 애매한 종목(6.0점)은 자연히 걸러진다. "단정" 대신 "강하게 눌러주기" — 첫 설계를 고집하지 않고 근거 있는 피드백으로 구조를 바꾼 경험.
결정 04 · 정책 단순화

안정형 제거, 공격형 단일 + 실험용 프리셋 3종

문제
처음엔 안정형/공격형 2개 정책을 이원화해서 모든 문턱을 두 벌로 관리했다. 복잡도가 두 배.
고민
MVP에서 두 정책을 다 검증할 수 있나? 그리고 문턱 숫자(0.60, 5.0…)는 어차피 가설인데, 하나로 고정해도 되나?
결정
공격형 하나로 단순화하되, 그 안에서 강도별 프리셋 A(신중)/B(표준)/C(초공격)를 만들어 백테스트로 비교. "정답 하나"가 아니라 "기대값(승률×수익 − 패률×손실)이 최대인 조합을 실험으로 찾는다"는 접근으로 바꿨다.
결정 05 · 에이전트 간 협상

Critic에게 뭘 보내나 — 설득 payload와 역할 경계

문제
내 다음 단계는 반박 전문 Critic. "매수 8.5"만 던지면 "근거는? 위험은 생각했어?"로 기각된다.
고민
그럼 약점(bear case)까지 전부 내가 찾아서 줘야 하나? → 아니다. 반박 논리를 총동원하는 건 Critic의 일이지 내 일이 아니다. 역할이 섞이면 견제 구조가 무너진다.
결정
근거(bull_case) + 내가 인지한 핵심 위험 1개(key_risk) + 반박 대비 + 대응 시나리오 + 숫자 증거까지만. 원본 신호는 통째로 안 넘기고 열쇠(cycle_id+ticker)만 줘서 Critic이 직접 검증하게 했다. "위험을 인지했다는 신뢰 표시"와 "Critic 일 대신하기"의 경계를 그은 것.
결정 06 · 문서화 방식

팀 공유 문서를 "살아있는 웹사이트"로

문제
설계가 계속 바뀌는데(거부권 폐지, 정책 단일화…) 매번 파일을 다시 돌리면 팀원마다 다른 버전을 보게 된다.
고민
"최신 상태 하나의 주소" + "과거 버전 추적"을 동시에 하려면?
결정
GitHub Pages로 설계 문서를 배포하고, 모든 변경을 git 태그(v1.0~v2.1)로 버전 박제. 팀원은 항상 같은 URL에서 최신을 보고, 과거 결정은 태그로 돌아가 확인한다. 문서에 수정 시각·변경 이력도 명시.
Deliverables

산출물 — 내가 만든 것들

📐 판단 로직 설계 (11단계)

hard_block → 신호 4종 카운트(AND 조건) → consensus 합산 → 교차확인 보너스(+0.15) → 매크로 감점 → GPT-4o 판단 → min_conviction 검증 → 저장·전달. 각 단계의 기준·처리·이유를 전부 문서화.

🤝 인터페이스 계약 (다이어트 JSON)

4개 분석팀의 전체 스키마에서 내가 실제 쓰는 필드만 추린 요청 스펙. ✅필수/📄참고/❌불필요를 구분해 팀 간 개발 결합도를 낮춤.

🗄️ ERD + 컬럼 사전

읽는 테이블 5개(신호 4종 + 메모리) & 쓰는 테이블 1개(tb_strategist_signals, 16컬럼)의 구조·타입·관계 설계. cycle_id + ticker를 공통 열쇠로.

🎯 정책 프리셋 3종 (실험 설계)

공격형 A(신중)/B(표준)/C(초공격) — 문턱값 세트를 3벌로 만들어 백테스트로 기대값 최대 조합을 선택하는 실험 프레임.

🧠 GPT-4o 판단 프롬프트 구조

신호 요약 + 판정 결과 + consensus + 과거 메모리 5건 + 정책을 넣고, 채점 루브릭 기반 conviction(0~10)을 받는 구조. GPT는 계산하지 않고 맥락만 판단.

🌐 설계 문서 사이트 (14개 섹션)

GitHub Pages 배포, 목차 내비게이션, 라이트/다크 모드, git 태그 11개(v1.0~v2.1)로 전체 의사결정 이력 보존.

Learnings

이 프로젝트에서 배운 것들

  • LLM 시스템은 "울타리 설계"가 절반이다.AI에게 뭘 맡기고 뭘 코드로 강제할지 경계를 긋는 게 프롬프트보다 먼저였다. 절대 규칙(상폐 차단, 확신 문턱)은 Python이, 맥락 판단은 GPT가.
  • 첫 설계를 고집하지 않는 것도 실력이다.매크로 거부권은 내가 공들인 설계였지만, "경기가 나빠도 오르는 종목이 있다"는 팀원 지적이 맞았다. 근거가 타당하면 구조를 다시 짠다 — 거부권 → 감점 전환.
  • 숫자 문턱은 "정답"이 아니라 "가설"이다.0.60, 5.0 같은 값은 논리로 정한 초기값일 뿐. 프리셋 3종을 만들어 기대값 기준으로 실험·선택하는 구조로 바꿨다.
  • 에이전트 협업은 역할 경계가 생명.Critic을 설득하려고 약점까지 다 내가 찾아주면 견제 구조가 무너진다. "인지한 위험 표시"까지가 내 몫, "반박 총동원"은 Critic 몫.
  • 인터페이스 계약이 팀 개발 속도를 정한다."내가 이 필드만 쓴다"를 먼저 문서로 못 박으니, 4개 팀과 내가 서로 기다리지 않고 병렬로 개발할 수 있게 됐다.
  • 문서는 파일이 아니라 "주소"여야 한다.설계가 바뀔 때마다 같은 URL에서 최신을 보게 하고, git 태그로 과거 결정을 추적 가능하게 — 문서 자체를 버전 관리 대상으로.
